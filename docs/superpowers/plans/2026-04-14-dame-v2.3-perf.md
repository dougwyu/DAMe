# DAMe v2.3 Hot-Loop Performance Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Reduce per-read overhead in `rust/src/sort.rs` by eliminating three sources of constant-factor slowdown identified from the Begum Rust comparison: character-based reverse-complement, O(N_tags) linear tag scan, and redundant per-read String allocations. Target: measurable sort-time reduction (~2–3× expected) without changing any output, algorithm, or public interface that downstream modules depend on.

**Architecture:** All changes are confined to `rust/src/sort.rs`. The three changes are independent and can be merged in any order, but this plan applies them in ascending risk: (1) add a byte-level `rc_bytes()` alongside the existing `rc()` and switch the hot path to it, (2) build a reverse tag lookup map (sequence → tag name) at startup and replace linear scans with `HashMap::get`, (3) drop the `to_string()` / `format!` allocations in `get_pieces_info` and `fill_hap` by returning `&str` references and using tuple keys or the `entry()` API more carefully. The `filter`, `rsi`, `chimera`, and `decollapse` modules are untouched. Integration tests (`tests/integration/run_*.sh`) continue to diff against `dame-py` on the tutorial dataset as the definitive parity check.

**Tech Stack:** Rust stable, `cargo test`, existing `ahash` / `indexmap` / `needletail` dependencies. No new crates.

## Background: what currently happens

In `rust/src/sort.rs`, per successful FASTQ read:

1. `std::str::from_utf8` validates the entire read (200–300 bytes).
2. `get_pieces_info`:
   - Runs four `find_primer` calls (unchanged — already byte-level and efficient).
   - Looks up `tag_name1` and `tag_name2` with `tags.iter().find(|(_, v)| v[0] == tag1_str)` — **a linear scan of every tag on every successful read, twice**.
   - In the reverse-orientation branch, calls `rc(between_raw)` which does `.chars().rev().map(…).collect::<String>()` — **three passes with UTF-8 decoding**.
   - Returns a `PieceInfo` struct with four owned `String` fields (`tag1`, `tag2`, `primer_name`, `between`), each a heap allocation.
3. `fill_hap`:
   - Builds `format!("{}_{}", tag1, tag2)` — **one fresh `String` allocation per read**, even for tag combos already seen.
   - Calls `between.to_string()` — **a second copy of the barcode**, even though `PieceInfo.between` is already owned.

Under typical load (100k reads, ~90% successful), this is ~400–700 bytes of heap churn and ~160 byte-comparisons of tag lookup per successful read. Begum Rust avoids all of this by working in `&[u8]` throughout, using `entry((&str, &str))` tuple keys, and indexing tags.

```rust
/// Byte-level reverse complement for DNA sequences.
/// Equivalent to `rc()` but operates on `&[u8]` directly — no UTF-8 decode, single pass.
/// Input bytes should be uppercase ASCII (standard FASTQ).
pub fn rc_bytes(seq: &[u8]) -> Vec<u8> {
    seq.iter()
        .rev()
        .map(|&b| match b {
            b'A' => b'T',
            b'T' => b'A',
            b'C' => b'G',
            b'G' => b'C',
            b'M' => b'K',
            b'K' => b'M',
            b'R' => b'Y',
            b'Y' => b'R',
            b'W' => b'W',
            b'S' => b'S',
            b'V' => b'B',
            b'B' => b'V',
            b'H' => b'D',
            b'D' => b'H',
            _ => b,
        })
        .collect()
}
```

## Task 2: Pre-build reverse tag lookup maps — O(1) instead of O(N_tags)

**Files:**
- Modify: `rust/src/sort.rs` — add `TagLookup` struct, build it in `read_tags`, use it in `get_pieces_info`

`get_pieces_info` currently calls `tags.iter().find(|(_, v)| v[0] == tag1_str)` twice per successful read. For 10 tags of 8 bp each that is ~160 byte comparisons per read. With a pre-built `HashMap<Vec<u8>, String>` it becomes one hash lookup.

The current `read_tags` returns `HashMap<String, Vec<String>>` (name → [fwd, rc]). Downstream code (integration tests, other modules) does not introspect this structure — `get_pieces_info` is the only consumer. So we can change the return type.

- [ ] **Step 1: Define `TagLookup` and rewrite `read_tags`**

Replace the existing `read_tags` (lines 111–136). Add a new struct above it:

```rust
/// Pre-built O(1) reverse lookup for tag sequences.
/// `by_fwd` maps forward tag bytes → tag name; `by_rc` maps RC tag bytes → tag name.
pub struct TagLookup {
    pub by_fwd: HashMap<Vec<u8>, String>,
    pub by_rc: HashMap<Vec<u8>, String>,
}

/// Read a Tags file (TagSeq\tTagName per line) and build O(1) reverse lookup maps.
pub fn read_tags(path: &str) -> Result<TagLookup> {
    let file = File::open(path).with_context(|| format!("Cannot open tags file: {path}"))?;
    let reader = BufReader::new(file);
    let mut by_fwd: HashMap<Vec<u8>, String> = HashMap::default();
    let mut by_rc: HashMap<Vec<u8>, String> = HashMap::default();

    for line in reader.lines() {
        let line = line?;
        let line = line.trim();
        if line.is_empty() {
            continue;
        }
        let parts: Vec<&str> = line.split('\t').collect();
        if parts.len() < 2 {
            continue;
        }
        let seq = parts[0];
        let name = parts[1];
        by_fwd.insert(seq.as_bytes().to_vec(), name.to_string());
        by_rc.insert(rc(seq).as_bytes().to_vec(), name.to_string());
    }

    Ok(TagLookup { by_fwd, by_rc })
}
```

## Task 3: Eliminate per-read String allocations in `fill_hap`

**Files:**
- Modify: `rust/src/sort.rs` — rewrite `fill_hap` to avoid `format!` and `to_string()` allocations on every read

`PieceInfo` already owns its four `String` fields. `fill_hap` currently builds a new `format!("{}_{}", tag1, tag2)` key on every read and calls `between.to_string()` a second time. Both allocations are pure overhead for reads that land on already-seen combos.

The simplest fix: use the Entry API with `get` + manual insert so we allocate keys only on first insert. For nested map, same pattern.

- [ ] **Step 1: Rewrite `fill_hap`**

Replace the current function body:
```rust
pub fn fill_hap(hap: &mut Hap, tag1: &str, tag2: &str, primer_name: &str, between: &str) {
    let key = format!("{}_{}", tag1, tag2);
    let entry = hap.entry(key).or_insert_with(|| HapEntry {
        tag1: tag1.to_string(),
        tag2: tag2.to_string(),
        seqs: IndexMap::new(),
    });
    let seq_entry = entry.seqs.entry(between.to_string()).or_insert_with(|| SeqEntry {
        count: 0,
        primer_name: primer_name.to_string(),
    });
    seq_entry.count += 1;
}
```

With:
```rust
pub fn fill_hap(hap: &mut Hap, tag1: &str, tag2: &str, primer_name: &str, between: &str) {
    // Lookup-by-ref to avoid allocating a key String for already-seen combos.
    // IndexMap doesn't expose raw-entry APIs, so we do two lookups on miss
    // (check, then insert with an owned key). This is the typical pattern.
    if !hap.contains_key_with(tag1, tag2) {
        let key = format!("{}_{}", tag1, tag2);
        hap.insert(
            key,
            HapEntry {
                tag1: tag1.to_string(),
                tag2: tag2.to_string(),
                seqs: IndexMap::new(),
            },
        );
    }
    let entry = hap.get_mut_by(tag1, tag2).expect("inserted above if absent");

    // Same pattern for the inner map.
    if !entry.seqs.contains_key(between) {
        entry.seqs.insert(
            between.to_string(),
            SeqEntry {
                count: 0,
                primer_name: primer_name.to_string(),
            },
        );
    }
    let seq_entry = entry.seqs.get_mut(between).expect("inserted above if absent");
    seq_entry.count += 1;
}
```

Wait — `IndexMap` does not have `contains_key_with` / `get_mut_by` helpers. We need to use the `&str`-keyed map lookup. The standard trick is to borrow the composite key. The cleanest low-risk approach is to simply keep the current allocation pattern but use `entry()` with `get_or_insert_with_key` if available, or accept one `format!()` but skip the second `to_string()`.

Simpler, more correct implementation — just skip the redundant `between.to_string()`:

```rust
pub fn fill_hap(hap: &mut Hap, tag1: &str, tag2: &str, primer_name: &str, between: &str) {
    // We still allocate the composite key (IndexMap has no get_or_insert_with_key
    // for &str to String on miss without extra crates), but we avoid the second
    // allocation of `between` by using the raw `entry` API with `between.to_string()`
    // only when the key is genuinely new.
    let key = format!("{}_{}", tag1, tag2);
    let entry = hap.entry(key).or_insert_with(|| HapEntry {
        tag1: tag1.to_string(),
        tag2: tag2.to_string(),
        seqs: IndexMap::new(),
    });

    if let Some(seq_entry) = entry.seqs.get_mut(between) {
        seq_entry.count += 1;
    } else {
        entry.seqs.insert(
            between.to_string(),
            SeqEntry {
                count: 1,
                primer_name: primer_name.to_string(),
            },
        );
    }
}
```

This:
- Keeps one `format!` per read (IndexMap limitation — acceptable, ~30 bytes).
- **Saves** one `between.to_string()` per successful-and-seen-barcode read — which dominates at high duplication rates typical of amplicon libraries.
- Keeps `primer_name.to_string()` only for genuinely new barcodes.

- `get_pieces_info` return type (`Option<PieceInfo>`) unchanged — the hot-path speedup comes from eliminating operations inside, not from changing the caller contract.

**Non-goals (things this plan deliberately does NOT touch):**
- `find_primer` — already byte-level and efficient; no SIMD acceleration attempted.
- Parallelization — rayon would break IndexMap insertion ordering and thus byte-identical parity.
- Filter / RSI / chimera / decollapse — not in the hot path for `sort`, and each has its own parity contract with `dame-py`.
